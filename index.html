01 Campus Apply Skill

01校园招聘投递辅助 Codex Skill

这个项目做什么

把校园招聘投递流程封装成一个 Codex Skill：读取结构化简历库、扫描表单字段、自动填写确定性事实，并停在登录、附件上传和最终提交等安全断点前。

Codex SkillHuman-in-the-loopResume YAML

用户痛点

  • 表单字段重复填写，系统差异大。
  • PDF 简历解析经常错位或缺失。
  • 提交动作不可逆，不能盲目自动化。
→

产品方案

  • 确定性事实自动填充。
  • 主观选择和高风险动作交给用户。
  • 每次投递后沉淀站点规则和缺失信息。

01 Campus Apply Skill

01产品判断与技术问题：效率提升不能牺牲用户控制权

Skill 负责

  • 读取简历库
  • 扫描表单字段
  • 填充确定性事实
  • 生成报告

用户负责

  • 登录与验证码
  • 附件上传
  • 主观意愿选择
  • 最终提交

技术问题

  • 不同 ATS 表单 DOM 差异大
  • PDF 解析字段不可信
  • 动态新增经历段落
  • 文件上传不能误提交

系统沉淀

  • 站点字段规则
  • 稳定个人事实
  • 操作经验
  • 下次交接文档

这个项目的核心不是“自动化到最后一步”，而是把高重复、低判断的动作交给 Skill，把高风险、强主观的动作留给人。

02 AI-KP Feishu TRPG Bot

02AI-KP 飞书群跑团主持机器人

这个项目做什么

把 AI 放进飞书群跑团场景：通过飞书机器人接收玩家行动，结合团本 RAG、骰点工具、场景状态和大模型生成，以 KP 口吻推进剧情。

Vertical AgentFeishu BotRAG / StateLLM Orchestration

  • AI 直接读团本容易剧透和前后矛盾。
→

产品方案

  • 玩家只需在群里 @AI-KP 描述行动。
  • 系统按 public / revealed / private 分层检索。
  • 场景状态和骰点结果进入大模型上下文。

02 AI-KP Feishu TRPG Bot

02当前如何跑通：飞书入口、规则工具与状态持久化

飞书接入

  • 自建应用开启机器人能力。
  • 长连接接收 im.message.receive_v1。
  • 解析 @ 消息后调用 reply 接口回群。

规则与状态

  • /r 支持骰点表达式和 COC 判定。
  • scenes.json 管理场景、出口、调查点。
  • session.json 保存当前场景。
  • revealed_notes 写回已公开线索。

技术难点

  • PDF 团本抽取后需要分块检索。
  • 玩家可见与 KP 私密资料必须隔离。
  • 自然语言“去二楼/调查壁炉”要映射结构化场景。
  • 多人群聊需要避免重复回复和机器人自触发。

跑通链路

本地启动 Python 服务后，飞书群 @AI-KP 触发消息事件；系统读取 campaign 资料、匹配当前场景、执行骰点或检索，再把回复发回原消息线程。

产品指标

开团准备时间、AI 平均回复延迟、单局人工接管次数、剧情一致性评分、剧透事故次数、玩家满意度与完团率。

这个项目的重点不是“聊天”，而是把 AI 放进一个有专业资料、工具、状态和权限约束的主持系统里。

03 My Keep

03My Keep｜个人健身记录与复盘工具

这个项目做什么

围绕个人健身复盘，把今日训练、月视图、维度数据和知识库放进同一个工作台，帮助用户从“记录动作”走到“看见变化并调整计划”。

Local DataMobile FirstAnalytics

用户痛点

  • 训练记录、身体数据、照片和知识散在多个工具。
  • 训练中录入重量、次数、左右侧数据不够顺手。
  • 记录后缺少可复盘的趋势视图。
→

产品方案

  • 今日页负责快速记录。
  • 月视图聚合训练频率和容量。
  • 维度数据和知识库帮助调整下一轮训练。

03 My Keep

03设计结构：今日输入、月视图复盘、维度分析、知识反馈

Today

今日

训练时只保留必要输入：动作、组数、重量、次数和完成状态，减少打断。

Calendar

月视图

把每天的训练部位、频率、容量和有氧记录聚合到一个月历里。

Metrics

维度数据

从训练容量、围度、体重、照片和频率拆出可观察指标。

Knowledge

知识库

把复盘结论沉淀为训练原则和下次计划，形成调整闭环。

可验证指标

单次训练记录耗时、每次记录点击数、连续记录天数。

技术问题

动作组需要支持左右侧、重量、次数、完成状态和容量计算，不能因为输入顺序不同导致统计错乱。

数据设计

训练记录、身体指标、照片和动作库拆成独立数据表，再按日期聚合到月视图复盘。

05 酒瘾人格测试

05酒瘾人格测试｜朋友小圈层娱乐测试

这个项目做什么

用户痛点

  • 朋友之间需要“有梗但不冒犯”的标签。
→

产品方案

  • 10 题短链路完成。
  • A/B/C 分值映射到 11 个酒类人格。
  • 结果页提供维度、酒搭子和截图分享点。

05 酒瘾人格测试

05轻量评分模型与传播闭环

评分模型

  • 10 道题，降低完成门槛。
  • A/B/C 分别映射 0/1/2 分。
  • 总分映射到 11 个酒类人格结果。
  • 结果页包含六维度、酒局表现和最佳酒搭子。

传播闭环

  • 朋友群或朋友圈看到链接。
  • 短题量快速完成。
  • 结果页提供标签、维度和最佳酒搭子。
  • 截图后在小圈层二次讨论。

技术问题

  • 结果不能纯随机，需要稳定可解释的映射。
  • 题目权重影响人格分布，要避免结果过度集中。
  • 移动端结果页需要适配截图分享。
  • 本地静态站点要保证无后端也能完整运行。
